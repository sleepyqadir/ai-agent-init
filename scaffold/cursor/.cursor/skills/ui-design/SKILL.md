# UI Design

Build UI that looks like it was designed by a human with taste, not generated by a pattern-matcher.

## Before Writing Any Code

Answer:
- What is the primary action on this page or component?
- Who is the user and what state are they in when they see this?
- What should they feel? (confident, focused, calm, excited?)
- What existing design system, tokens, or component library is in use?

Read 2–3 existing components to understand the visual language already established.

## Design Principles

**Hierarchy over decoration.** The most important element should visually dominate. If everything is emphasized, nothing is.

**Space is not empty.** Whitespace creates focus. Dense UIs are exhausting. Give elements room.

**Consistent rhythm.** Spacing, type scales, and border radii should follow a system — not be chosen arbitrarily per component.

**States are part of the design.** Loading, empty, error, and disabled states are not edge cases — they're the majority of what users see. Design them.

**Interaction communicates.** Hover, focus, and active states tell the user what's interactive. Don't skip them.

## Implementation

Phase 1: Structure — semantic HTML, correct ARIA roles, logical DOM order
Phase 2: Layout — grid/flex structure, responsive breakpoints
Phase 3: Typography — size scale, weight, line-height, color
Phase 4: Color — primary, neutral, semantic (success, error, warning)
Phase 5: Interaction states — hover, focus, active, disabled, loading, empty, error
Phase 6: Polish — shadows, borders, transitions (subtle — not decorative)

## Common Mistakes to Avoid

- Generic card-heavy layouts where every content type is a card
- Blue links on white backgrounds as the only interactive indicator
- Forms with labels above inputs but no visible grouping
- Empty states that just say "No data"
- Loading states that are just a spinner with no context
- Responsive design that's "fine on mobile" but clearly an afterthought

## Self-Check Before Submitting

- Does the hierarchy guide the eye to the primary action?
- Are all interactive states (hover, focus, disabled) implemented?
- Is the empty state designed, not just absent?
- Is the loading state designed?
- Does it look intentional on mobile?
